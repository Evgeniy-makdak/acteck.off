The concept of branching in Git

The most useful tools in Git arebranches (branches). They allow you to work in parallel on a project, switching between branches as necessary, solving various problems. Let's look at an example of what the essence of branches is and how they are used in practice.

So, as you already know, by default we have one branch -master. This branch is usually considered the main branch. However, you can create additional branches, as a rule, to solve specific problems of the project.

When creating a new branch, it will contain copies of our project files. We will write code in them, and then when the task is completed, wesaltcode for a new branch with our main branch. This way, you can work on several problems in parallel, so that they do not interfere with each other. And, as soon as you are ready, send the solved problems to production (that is, to users for use).

At the same time, when switching between branches, Git itself will change the files in the folder with your project. That is, some files will disappear, while others will appear. The texts of the files will also change if they differ from branch to branch. This is an important point, but newbies don't always understand it. Therefore, once again: when you change the branch in the terminal, your files and the texts in them will change in accordance with the branch. Physically! Sami! Right in the open editor, the text of your file will change to another! It's very cool and convenient.

Let's look at an example of how this is done in practice. Suppose you have some project. In the threadmasterthere is its current combat version, which is used by users. Let's say you're developing a new feature. To do this you created a new branchfeature. In it you write code to implement the task you need. Suddenly the phone rings and your boss says that a serious bug has been discovered in the project and it urgently needs to be fixed.

To do this you create a branchfix, copying the entire project code from the branch into itmaster. You write code that fixes a bug, and thenpour inthis code to the main branchmasterand send this code to production.

Then switch to the branchfeatureand calmly continue working on your feature. When it's finished, you merge it into the main branchmaster.
