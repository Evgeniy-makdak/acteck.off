Easily rebase commits in Git

Git provides two ways to make changes when working with branches:mergerAndrelocation.

The advantage of rebasing is that the commit history of both branches, unlike a merge, will look more linear and easier to understand for team members working on the same project.

In Git, merging done using the commandmerge, occurs as a three-way merge of the latest changes of the two merged branches and the newest snapshot of the parent branch shared by these merged branches. In this case, only the final snapshots of branch commits are merged, and not their entire previous history.

Rebasing involves applying changes from one branch (the current one) on top of another branch (which is merged into the current one). Let's create a branchtest, in which we will make a new filefile.txt, index it and commit it:

git branch test git switch test git add file.txt git commit -m "commit from branch test"

Now let's switch to the branchtestand rebase it relative to the main branchmaster:

git checkout test git rebase master

The rebasing process itself works like this: first, Git takes a snapshot of the parent branchmaster, which is saved in a temporary file. Then the current branchtestinserts its changes after the most recent commit of the branchmaster. Now the commit history of the main branch will look like this:

After a rebase, the branch's commit historytestwill also look similar to a branchmaster.

Then this can be moved to the branchmasterand perform the merge by rewinding, i.e.masterwill become master again and include all changes from the branchtest:

git merge test
