Merging branches in Git

As a rule, new branches are created as branches of the main branch to solve some problem. After the problem is solved, we canpour inthe contents of the side branch into the main one. This is done using the commandmerge.

Let's look at this process in more detail. Let us have a branchmasterand branchtest.

Let's switch to the branchtest, make changes to the file, index it and save the commit:

git switch test git add file.txt git commit -m 'commit from test'

After that we will return to the threadmaster:

git switch master

Now let's add it to our threadmasterthread contenttest:

git merge test

After running this command, Git will ask us to enter a merge comment. After entering the comment, a message indicating a successful merge will appear in the terminal.

Moreover, after merging the branchtestwon't disappear. Its contents will be merged into the main branch. The master branch will change after the merge, but the branchtestwill remain the same. We can continue working on the branchtest, periodically pushing its contents to the main branch.

It is important to pay attention to the terminology here. In literature and in life we ​​talk aboutdraintwo branches or aboutmergerbranches. And when we say this, it seems that there were two different branches and they merged into one - a common one. In fact, we are not merging branches, butpour inthe contents of one branch to another. They don’t say that in life (and you don’t say it), but always understand the essence of the process - we pour in, not drain.
