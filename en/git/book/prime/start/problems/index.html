Backup problem

Imagine that you have some big project. You write it, at some point you reach some logical completion and roll out a version1.0, which users are starting to use.

The work, however, does not end there and you roll out the version1.1with bug fixes and additions, then a version1.2, Then1.3, and then you rewrite everything again and announce the release of the version2.0. And so on - work on a project usually never ends.

During the development process, however, there is a risk of breaking something in the code, or accidentally deleting the project or part of it. To avoid losing all your work, you periodically save the project code in a separate folder. You simply number your saves in order, indicating some comment on the save about what changes you have made or what functionality you have already done at this point in saving.

However, it suddenly turned out that you can’t just roll out the version2.0, because users of the first version are indignant - they are not yet ready to switch to the second. Okay, you say, and start working on the second version, while simultaneously releasing updates to the first. Your project is divided into two folders - for the first version and for the second. The backup folder also collapses. Moreover, some of the code of the first and second versions are the same. When you make changes to this code in one version of the project, you must remember to make changes in the other version. However, you sometimes forget to do this. Strange bugs appear. The work is worth it.

Version in progress2.0you decide to add a new feature. However, it takes a long time to do it and you decide to split the second version into two folders, duplicating the code. In the first folder you will conduct further development on your project, and in the second you will create an invented feature. Finally, the feature is completed. During this time, the main project has already become a version2.3. Now you need to merge the code from the feature with the main project code. You start copying files with code into the main folder, simultaneously resolving conflicts between the new code and the old. This takes you three working days. The work is worth it.

The project has grown. You decide to call a friend for help. You give a friend a task and send him an archive with the project. A friend completes the task and returns you an archive with new files, as well as old ones to which he has made changes. You spend half a day separating new files from old ones. Then you spend another day integrating the changes into your files.

You call two more friends for help. Now there are four of you. Every day you are sent archives of completed work. You integrate new parts into the project and send out project archives with changes to everyone. The work progresses slowly, because while you are integrating one change, your friends are waiting for you to send them a new version of the project and cannot work further, lest they become completely out of sync.

The backup folder already contains gigabytes of code. At the same time, there are mostly duplicates there, because if you change a small part of the code, you still reset the entire project. You decide to clean out this folder by deleting the first hundred versions. A week later, it turns out that the deleted versions contained code for a separate feature, which you then abandoned, and now decided to return. But the code with this feature has already been deleted and is nowhere to be found. You write this feature again. The work is worth it.

Doesn't sound like much, right? However, the developers actually used to work as I described above. This, of course, was very inconvenient. Therefore, a special Git system appeared, the capabilities of which we will talk about in the next lesson.
