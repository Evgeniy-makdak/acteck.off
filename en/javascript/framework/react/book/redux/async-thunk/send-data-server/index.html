Handling a POST request sent using thunk in Redux

We received the data of products and sellers from the server and displayed them in the application. But with thunk you can not only receive, but also send data. Let's find out how to do this.

After we add a new product, it remains in our store, that is, inside our application. Let's make sure that the new product is saved on the server.

Let's start with the server. Here we will have to process not a GET, but a POST request, since the server will now receive product data, which we will save there.

Let's open our application with products, and in it there is a fileserver.js. Let's find an array in ithandlersand add another handler for the POST request to it. Here we will also receive the request body, so we must pass in the callbackrequest:

http.post('/fakeServer/products', async ({ request }) => {})

Now we will write the code of our callback in curly brackets. First, we will extract the request data and, in case of an error, send a response from the server that we could not save the data and status500:

http.post('/fakeServer/products', async ({ request }) => { const data = await request.json() if (data.content === 'error') { await delay(ARTIFICIAL_DELAY_MS) return new HttpResponse('server save error', { status: 500, headers: { 'Content-Type': 'application/json', }, }) } })

If everything is in order with the data, then we will find the seller in the database by the id that came to us in the request and enter this seller intodata(after all, we also store the seller’s data on the server ;) ):

const seller = db.seller.findFirst({ where: { id: { equals: data.seller } }, }) data.seller = seller

Next, we will create an object with reactions for this product in the database. And now, having all the necessary fields for the product, let’s create the product itself in the database:

data.reactions = db.reaction.create() const product = db.product.create(data)

Let's set a delay and return a response with a product in the last line of code for our callback:

await delay(ARTIFICIAL_DELAY_MS) return HttpResponse.json(serializeProduct(product))

At this point, we are completely done with the server and will not return to it again.

By the way, another useful thing. After the lineexport const worker = setupWorker(...handlers)you can add the following code:

worker.listHandlers().forEach((handler) => { console.log(handler.info.header) })

And now you can see the result of firing each handler in the browser console.

Of course, our server is not real and if we forcefully refresh the page in the browser, then all our new product objects will disappear.
