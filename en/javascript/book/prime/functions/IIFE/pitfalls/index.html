IIFE Pitfalls in JavaScript

Let's look at two pieces of code.

First:

let result = 1 +function() { return 2; }(); console.log(result);

Second:

let result = 1; +function() { return 2; }(); console.log(result);

These two pieces of code are almost the same, but if you run them, the result will be different. The first code will print a number to the console3, and the second is a number1.

Why is there such a difference: the whole point is that in one case there is no semicolon at the end in the first line of code, and in the second case there is a semicolon.

You may ask: how can this be, since in JavaScript the semicolon at the end of the command is optional! In fact, this is not entirely true. Let's figure out what's really going on here.

The first code can be rewritten like this:

let result = 1 + function() { return 2; }(); console.log(result); // выведет 3

Now it immediately becomes obvious that the result of calling the function in place is added to one, that is2. Therefore the final result will be3.

If you put a semicolon after the unit, the code will be interpreted differently by the interpreter:

// Первая команда: let result = 1; // Вторая команда: +function() { return 2; }(); // Третья команда: console.log(result); // выведет 1

That is, assigning to a variable and calling a function in place will become different commands. And all because of the presence of a semicolon!

It turns out that in this case calling the function in place does nothing at all - it just returns a number to nowhere2, which has no effect on the variableresult.

Let's then figure out why we can't write semicolons in JavaScript at all. Let's say we have code like this without semicolons:

let result = 1 // в result запишется 1 let test = 2 // в test запишется 2

It works correctly because the interpreter itself placed a semicolon at the end of each line.

But look at this code:

let result = 1 + 2; // в result запишется 3

Now the semicolon at the end of the first line will not be placed automatically, since the interpreter understands that the second line command is part of the first line command.

But if we put a semicolon ourselves, the result will be completely different:

let result = 1; // в result запишется 1 + 2; // команда ничего не делает, но и ошибки не будет

It turns out that the interpreter itself puts a semicolon only if the next command is not part of the previous one.

Now look at this code:

let result = 1 +function() { return 2; }(); console.log(result);

Indeed, the second line is simply a continuation of the command on the first line and the interpreter does not automatically insert a semicolon. That is why, if we ourselves write a semicolon at the end of the first line, the result will be completely different. This suggests thatIt's best to always put semicolons in the right places to avoid problems.
