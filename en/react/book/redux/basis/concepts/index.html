Two basic ideas in Redux

Before we start exploring the Redux library, let's get acquainted with its two basic ideas. The first idea regardingstate management, we will look at the example of a regular counter on a state in React:

function Counter() { // State const [count, setCount] = useState(0) // Action: function clickHandler() { setCount(count + 1) } // View: return ( <div> <span>{count}</span> <button onClick={clickHandler}>+</button> </div> ) }

In this code we see the state for the variablecountis the source of truth for the counter. Then we seeaction (action) - an event that, when clicked by a user, causes a state update. And finallyperformance (view), with which we display the user interface.

Thus, the scheme is quite simple: the button is pressed - statecount(State) when clicked (Action) changes, therefore the view (View) changes - in our case, this is the number on the screen.

However, things will be more complicated if we have many components that must use the same state. It is possible, of course, for exampleraise your fortuneto parent components, but this solution may not always solve the problem. Redux in this case suggests that we createone place in the application that would contain the global stateand certain patterns of behavior when updating this state. This is precisely the first basic idea (!).

The second idea is related toimmutability, that is, with data immutability. You, of course, remember from JavaScript that arrays and objects can be modified. Changing data directly in Redux, as in other frameworks, is considered bad manners and can lead to unpredictable consequences. Therefore, in Redux, our original objects and arrays must remainimmutable, and we can only change their copies.
