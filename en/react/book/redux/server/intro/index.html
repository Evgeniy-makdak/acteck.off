Communication between Redux application and server

This section will be for experienced students who did not stop there with Redux and decided to reach the end of this tutorial. In the next two sections, we'll touch on how a Redux app works with asynchronous logic, connecting our app to the backend and teaching it how to communicate with it.

From the previous sections, we know that the synchronous mechanism is at the heart of a Redux application. The main stages of work are to send actions when some event occurs in the application, then update the state in the store with a reducer function, notify application components about the state change and, accordingly, display the new state value in the components.

In real life, it often happens that an application also needs asynchronous logic, for example, when working with HTTP API requests and receiving data from some external server. Therefore, in order for our'синхронный'Redux store could work with such asynchronous logic using the so-calledmiddlware- software that is an intermediary or connection between the store of our application and, for example, the server with which we will begin working in this chapter of the tutorial.

Many such middleware have been created for Redux, but the most widespread among them and officially recommended when running Redux applications with asynchronous logic is stillRedux Thunk. This software helps us use regular functions (or thunks) that can contain asynchronous logic such assetTimeout, Promises, async/await, to interact with our store. As for the term itself,thunkis code that does some deferred work.

Now that we've learned a little about how we can work with asynchronous logic in Redux, we can start implementing this step in our application. We need to make sure that our application can interact with some external server, receiving data from it and transmitting it to it. In other words, there is an exciting and very important topic ahead of us. Let's begin.
