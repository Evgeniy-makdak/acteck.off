Bypassing the limitation on the number of databases on hosting

As you already know, the number of databases on shared hosting is usually limited. As a rule, it is equal to the number of sites.

However, there are situations when we need more databases. Let's discuss what these situations are and how to bypass the limitation on the number of databases.

So. Let us have the cheapest tariff, in which we can make only one website and, accordingly, one database.

However, we can make one websitesecondlevel, but any number of subdomains. For example, we can have an online store on the main domain, and a forum on the subdomain.

It turns out that we need two databases. And only because of this we will have to take a more expensive tariff. But there is a workaround.

The essence of the workaround is as follows. Both of our sites will live in the same database. But to prevent table names from conflicting, we will give them prefixes.

For example, we will start the names of the store tables withshop_, and the names of the forum tables are onforum_. However, usually for brevity prefixes are made with two letters, in our case it could besh_Andfr_.

Of course, changing table names throughout your code can cause some inconvenience. Therefore, when developing a website, it is better to immediately anticipate the possibility of such a situation and not hardcode table names, but store them somewhere in one place so that they can be easily changed.

Or we can immediately provide for the presence of a prefix in our code and add a prefix before the table name everywhere in SQL queries. The prefix itself will be stored somewhere in one place - in the configuration of our site. This way we can easily fix it to the one we need.

Consider the code you will write in your project to support table prefixes.
