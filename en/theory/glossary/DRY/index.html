DRY programming principle

Programming principleDRY (Don’t repeat yourself) involves breaking down a large system, such as software you've developed, into smaller, non-repetitive components. If you have several components that perform the same tasks, then the DRY principle should reduce the number of components, ideally so that each component is not repeated.

Once the system has been divided into components responsible for performing clearly defined tasks, they can be organized into classes, which is calledmodular architecture.

To properly build a system based on the DRY principle, you must adhere to the following rules:

  • Before you start working on a project, visualize it as a graphical diagram divided into visual components.
  • When working on a complex project component, it should also be represented graphically in the form of a UML diagram.
  • The graphical diagram should clearly indicate the hierarchy and role of each project component.
  • The diagram should also indicate the relationship of your components with the components of other project participants, as well as which branches of the project will be public or private.
  • It is necessary to avoid rigid connections between components, as they negatively affect the efficiency of the entire project architecture.
